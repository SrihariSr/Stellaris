# Project Stellaris

A deep learning pipeline for exoplanet candidate identification in NASA Kepler and TESS photometric data, demonstrating cross-mission generalisation from a Kepler-trained model to TESS without fine-tuning.

---

## What this does

Stellaris takes raw light-curve data from NASA's TESS mission, searches it for periodic transit-like signals, scores those signals with a Kepler-trained convolutional neural network, and produces a ranked list of planet candidates with confidence scores and false-positive vetting diagnostics.

The pipeline was trained on 6,726 Kepler Objects of Interest, then deployed unchanged across three different TESS regimes:

- **1,102 confirmed TESS planets** — for validation. Recovered 62% at CNN ≥ 0.5 + vetting passed.
- **4,002 unresolved TESS candidates** that the astronomical community is actively investigating — for ranking. Strongly endorsed 1,564 of them.
- **474 faint stars (Tmag 13-15)** not previously catalogued — for exploration. A single candidate cleared the CNN threshold and both vetting rules (TIC 155810890), and it has not yet been through the four-check screening. Its BLS period sits 5.04% from the 13.7-day TESS orbital systematic, just outside the 5% rejection tolerance, so it is most likely instrumental. The thin yield characterises the model's sensitivity floor in the faint regime.

## How it works

```
NASA Kepler / TESS FITS files
            ↓
Light curve extraction (PDCSAP flux via lightkurve)
            ↓
Quality masking + 5σ outlier clip + wotan median detrend (1.0 day window)
            ↓
Box Least Squares search across 50,000 trial periods (0.5–20 days)
            ↓
Phase-folded views: 2001-bin global + 201-bin local (depth-normalised to -1)
            ↓
StellarisNet two-branch CNN (9.7M params, PyTorch)
            ↓
Rule-based vetting: depth cap (3%) + odd/even consistency (15% max diff)
            ↓
Optional rigorous validation:
  • Secondary eclipse check (phase ±0.5, flags EBs at >3σ)
  • TESS systematic period check (rejects matches to 1d, 2d, 6.85d, 13.7d)
  • Period alias check (masks primary, re-runs BLS on residuals)
  • Sector consistency check (transit must reproduce across ≥2 sectors)
            ↓
Ranked candidate list with full diagnostics
```

## Quickstart

### 1. Build the training set

```bash
# Fetch the KOI catalog
python scripts/fetch_catalogs.py

# Download Kepler light curves (~41 GB, takes hours)
python scripts/fetch_kepler_lightcurves.py

# Build the HDF5 training set
python scripts/build_kepler_trainset.py
```

### 2. Train the model

```bash
python scripts/train.py
```

Outputs checkpoint to `checkpoints/stellaris_best.pt`. Final epoch: ~40, best val PR-AUC ~0.946.

### 3. Evaluate on held-out Kepler test set

```bash
python scripts/evaluate.py
```

Produces test PR-AUC, ROC-AUC, confusion matrix, and `results/test_evaluation.png`.

### 4. Run a hunt on TESS targets

```bash
# Build a target list (e.g., confirmed TESS planets)
python scripts/build_validation_list.py

# Run the hunt
caffeinate -i python scripts/hunting.py \
  --tic-list data/catalogs/tess_validation_confirmed.txt \
  --output results/tier1_results.csv \
  --failures results/tier1_failures.log
```

This processes each TIC: fetches TESS light curves from MAST, preprocesses, runs BLS, scores with the CNN, applies vetting. Output is a CSV with one row per target. Resumable via Ctrl+C (the CSV is the state).

### 5. Run rigorous validation

```bash
python scripts/validate_candidates.py \
  --results results/tier3_results.csv \
  --threshold 0.95 \
  --max-candidates 30
```

Runs 4 false-positive checks on the top candidates. Outputs per-candidate diagnostic plots in `results/validation/` plus a summary CSV.

### 6. Generate phase-fold diagnostics

```bash
python scripts/inspect_candidates.py \
  --results results/tier3_results.csv \
  --top 100
```

Produces phase-folded plots for visual inspection in `results/candidate_plots/`.

## Model architecture

**StellarisNet** is a two-branch 1D convolutional neural network with 9.7M parameters, following Shallue & Vanderburg (2018):

**Global branch** (input: 2001-bin phase-folded light curve):
- 5 ConvBlocks: Conv1d → BatchNorm → ReLU → MaxPool
- Channels: 1 → 16 → 32 → 64 → 128 → 256
- Flatten

**Local branch** (input: 201-bin zoomed view of transit):
- 2 ConvBlocks (same structure)
- Channels: 1 → 16 → 32
- Flatten

**Head**:
- Concatenate both branch outputs
- 3× Linear(512) with Dropout(0.5) in between
- Single sigmoid logit → planet probability

**Training**:
- BCEWithLogitsLoss with `pos_weight=1.462` (class imbalance correction)
- Adam optimizer, lr=1e-3, weight_decay=1e-4
- Dropout 0.5
- 40 epochs, best at epoch 33
- Train / Val / Test split by `kepid` (4,680 / 1,026 / 1,020)

## Validation framework

The validation script (`scripts/validate_candidates.py`) implements four false-positive checks beyond basic vetting:

**1. Secondary eclipse search.** Examines phase ±0.5 for a dip indicating an eclipsing binary's secondary eclipse. Flags at >3σ significance.

**2. TESS systematic period check.** Rejects candidates whose period matches known TESS instrumental periods (1d, 2d, 6.85d, 13.7d, 27.4d) or their integer multiples within a 5% tolerance.

**3. Period alias check.** Masks the primary signal and re-runs BLS on the residuals. If a comparably strong signal appears at a different period (ratio > 0.5), flags as ambiguous.

**4. Per-sector consistency.** Verifies the transit reproduces across multiple observed sectors. Single-sector signals get flagged as potentially one-off systematics or glitches.

**Important caveat:** This framework catches a meaningful subset of false positives but does NOT constitute statistical validation in the field's technical sense. Tools like TRICERATOPS or vespa compute False Positive Probability with explicit statistical models for blended eclipsing binaries, hierarchical triples, and background contaminants. The validation here is a useful pre-filter, not a replacement for proper community vetting.

## Limitations and honest caveats

This project is genuinely useful but has clear limits. Documented here for transparency:

- **No centroid analysis.** TESS pixels are 21 arcseconds wide and frequently contain multiple stars. A faint background eclipsing binary blended into a target's pixel can produce a diluted "transit" signal indistinguishable from a real planet by shape alone. Centroid analysis would catch these blends; this pipeline doesn't do it.

- **No statistical validation.** As noted above, the vetting is not statistical validation. To call any candidate a "validated planet candidate" in the TFOPWG sense would require additional rigorous statistical work.

- **No follow-up observations.** Confirming a planet requires radial velocity measurements or other independent observations. This pipeline produces candidates, not confirmations.

- **Eclipsing binary contamination.** The model was trained on Kepler data and learned transit shapes well, but cannot distinguish planets from eclipsing binaries on shape alone. TOI-2533, a confirmed EB, received CNN probability 0.988 in testing.

- **Sensitivity floor on faint stars.** The novel hunt on Tmag 13-15 targets produced one endorsed-and-vetted candidate from 474 successful targets, and that candidate has not been screened. Of the top eight faint-star signals by CNN score, six have BLS periods within 10% of the 13.7-day TESS orbital period, and five would be rejected outright by the systematic-period check. Across all 474 targets, 41.6% have a BLS period within 10% of 13.7 days. BLS is locking onto the TESS orbital systematic on faint stars, and the CNN cannot reject those signals because view normalisation makes it blind to depth. Fine-tuning on TESS would likely help.

- **Conservative validation checks.** The "TESS systematic period" check flags any candidate within 5% of known instrumental periods, which can reject real planets whose orbits happen to fall there. Similarly, single-sector candidates fail the sector-consistency check by default. These are conservative choices favouring false-negative reduction over false-positive avoidance.

**What this means for users:** Candidates surfaced by this pipeline are worth investigating, not declaring discovered. Treat the output as a prioritisation tool, not a discovery list.

## Citation

This project builds on foundational work in ML-based transit detection:

- **Shallue & Vanderburg (2018)**: "Identifying Exoplanets with Deep Learning." The original two-branch CNN architecture this work follows.
- **Princeton T16 (April 2026)** — Large-scale TESS ML candidate identification in faint regime.

## Reproducibility

All results in this repository can be reproduced from the scripts provided. Important reproducibility notes:

- **Random seeds** are set in training scripts (default: 42).
- **Train/val/test split** is deterministic, by `kepid` hash.
- **HDF5 training set** can be rebuilt from the KOI catalog with the provided scripts.
- **TESS hunt outputs** depend on MAST data availability at run time. Results may vary slightly if MAST data is updated.

Exact hyperparameters for the production model are in `scripts/train.py`. The checkpoint `checkpoints/stellaris_best.pt` was trained with:
- 40 epochs
- Dropout = 0.5
- Weight decay = 1e-4
- Learning rate = 1e-3
- Batch size = 32
- Seed = 42

## Scope and prioritisation

> *He who defends everything defends nothing.* — Frederick the Great

On a single-machine budget, ranking already-flagged TESS candidates and hunting new faint-star signals couldn't both be done at depth, so I prioritised the known-target work. The pipeline ranks 4,002 unresolved candidates and puts the strongest through four-check false-positive vetting; 1,564 are strongly endorsed (CNN ≥ 0.9) and 10 of the top 30 survive all four checks. The novel faint-star hunt was kept deliberately bounded (2,000 targets, 474 usable) and surfaced a single unscreened candidate, against 1,564 from the known-target stream. That thin yield is the informative result, and its cause is specific: BLS locks onto the 13.7-day TESS orbital systematic on faint stars, and the CNN cannot reject it. This is the case for the prioritisation.
